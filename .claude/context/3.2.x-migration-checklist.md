> 从 2.7.x 分支合并代码到 3.2.x 分支时，需要做以下适配。
> 3.2.x = Spring Boot 3.2.12 + Jakarta + JDK 17 + GraalVM 预备

## 4. Spring Framework 6.x API 变更

### 4.1 HttpMethod → 从 Enum 变为 Interface

| 变更 | 代码影响 |
|------|---------|
| `HttpMethod.resolve(String)` → `HttpMethod.valueOf(String)` | `MappingRegistry.java` 中调用点 |
| `EnumSet<HttpMethod>` → `Set<HttpMethod>` | `HttpMethodMatcher.java`、`HttpBodyCodecRegistry.java` |
| `EnumSet.copyOf(...)` → `new HashSet<>(...)` | `MappingRegistry.java` 中创建集合 |
| `httpMethod.name()` → `httpMethod.toString()` | `HttpMethodMatcher.java` |
| `switch(httpMethod)` → `if-else` 链 | `OpenApiAdapter.java` |

### 4.2 getBridgedMethod() → getMethod()

Spring 6 `HandlerMethod` 移除了 `getBridgedMethod()`，统一使用 `getMethod()`。

| 文件 | 修改 |
|------|------|
| `InvokableHandlerMethod.java` | 删除 `getBridgedMethod()` 覆盖方法 |
| `MappingRegistry.java` | `mappingContext.getBridgedMethod()` → `getMethod()` |
| `ReturnValueResolverRegistry.java` | 3 处调用 |
| `SpringHandlerMethodReturnValueHandlerAdapter.java` | 1 处调用 |
| `OpenApiAdapter.java` | `ctx.getBridgedMethod()` → `getMethod()` |

### 4.3 HttpStatus → HttpStatusCode

| 类 | 修改 |
|----|------|
| `BaseWebServerHttpResponse.java` | `setStatusCode(HttpStatus)` → `setStatusCode(HttpStatusCode)` |
| `ResponseStatusExceptionResolver.java` | `ex.getStatus()` → `ex.getStatusCode().value()` 然后 `HttpStatus.resolve()` |
| `InvokableHandlerMethod.java` | `getResponseStatus()` 返回 `HttpStatusCode` |

### 4.4 ServerHttpResponse 接口新增方法（如有用到）

Spring 6 `ServerHttpResponse` 可能新增了方法，需检查接口实现完整性。

### 4.5 检查清单
- [ ] 全局搜索 `HttpMethod.resolve(` → 改为 `HttpMethod.valueOf(`
- [ ] 全局搜索 `EnumSet<HttpMethod>` → 改为 `Set<HttpMethod>`
- [ ] 全局搜索 `EnumSet.copyOf` 涉及 HttpMethod 的 → 改为 `new HashSet<>`
- [ ] 全局搜索 `.name()` 涉及 HttpMethod 的 → 评估改为 `.toString()`
- [ ] 全局搜索 `switch` 涉及 HttpMethod 的 → 改为 if-else
- [ ] 全局搜索 `getBridgedMethod()` → 改为 `getMethod()`
- [ ] 全局搜索 `getBridgedMethod` 的覆盖声明 → 删除
- [ ] 全局搜索 `HttpStatus` 在 `setStatusCode` 中的使用
- [ ] 全局搜索 `getStatus()` 在 `ResponseStatusException` 上的调用

## 6. GraalVM native-image 支持

### 6.1 新建 native-image 配置文件

目录：`META-INF/native-image/io.github.springperf/spring-boot-starter-web/`

| 文件 | 内容 | 注意 |
|------|------|------|
| `reflect-config.json` | 所有框架类的反射配置 | ~353 行，需要反射的类 + 构造器/方法/字段 |
| `proxy-config.json` | 动态代理接口 | 11 个接口（包括 Spring Data、HttpMessageConverter 等） |
| `resource-config.json` | 资源模式 | spring 配置资源路径 |

### 6.2 代码层面的兼容

| 位置 | 变更 |
|------|------|
| `FastInvokerGenerator.java` | GraalVM 环境下抛出 `UnsupportedOperationException`，回退到 `MethodHandleInvoker` |
| `InvokableHandlerMethod.java` | 检测 native-image 环境，跳过 `FastInvokerGenerator` |

### 6.3 native-image 配置的维护

当新增框架组件类时，如果它被以下方式使用，需要加入 `reflect-config.json`：
- 通过反射创建实例（如 `WebComponentContainer` 中的工厂方法）
- 通过反射调用方法
- 通过反射访问字段

当新增动态代理接口时，需要加入 `proxy-config.json`。

### 6.4 验证测试

`GraalVmNativeImageConfigTest.java` 验证所有配置中的类确实存在于 classpath。

### 6.5 检查清单
- [ ] 新增组件类 → 检查是否需要加入 `reflect-config.json`
- [ ] 新增动态代理接口 → 加入 `proxy-config.json`
- [ ] 新增资源文件 → 检查是否需要加入 `resource-config.json`
- [ ] 运行 `GraalVmNativeImageConfigTest` 验证配置完整性
- [ ] 代码中涉及反射/字节码生成的地方需添加 GraalVM 兼容判断

## 10. 合并通用流程 Checklist

每次从 2.7.x 分支合并代码到 3.2.x 时，按顺序执行：

```
□ 1. 执行合并，解决 Git 冲突
□ 2. POM 检查
    □ Parent POM 版本正确？
    □ 新增依赖是否使用了 jakarta 坐标？
    □ 新依赖版本是否由 SB3 parent 管理？
□ 3. 全局搜索替换 javax.*
    □ javax.servlet → jakarta.servlet
    □ javax.validation → jakarta.validation
    □ javax.annotation → jakarta.annotation
□ 4. API 变更检查
    □ 搜索 getBridgedMethod() → 改为 getMethod()
    □ 搜索 HttpMethod 的 EnumSet/Enum/switch 用法
    □ 搜索 HttpStatus 在 setStatusCode 中的用法
    □ 搜索 getStatus() 在 ResponseStatusException 上的调用
□ 5. 自动配置检查
    □ 新增了 auto-configuration 类？→ 加入 AutoConfiguration.imports
    □ spring.factories 中新增了 EnableAutoConfiguration？→ 移到 .imports
□ 6. GraalVM 兼容检查
    □ 新增了反射调用的类？→ 加入 reflect-config.json
    □ 新增了动态代理？→ 加入 proxy-config.json
    □ 新增了字节码生成？→ 添加 GraalVM 守卫
□ 7. 构建 & 测试
    □ mvn install -DskipTests 编译通过
    □ 运行单元测试
    □ 运行集成测试
□ 8. 编译输出检查
    □ 确保 Java 17 语法兼容
    □ 确保无 javax.* 引用残留
```

## 11. Spring Boot 4.x（Spring Framework 7.x）兼容适配

> 本分支（3.2.x）需要同时兼容 SB 3.2.x（Spring 6.x）和 SB 4.0.x/4.1.x（Spring 7.x）。
> 通过 Maven Profile 切换版本：`spring-boot-4.0`（SB 4.0.7）和 `spring-boot-4.1`（SB 4.1.0）。
> 以下适配模式需在代码和测试中并存，不可使用条件编译。

### 11.1 HttpHeaders 不再继承 `Map<String, List<String>>`

**Spring 7.x 变更**：`HttpHeaders` 原来实现 `MultiValueMap<String, String>`（即 `Map<String, List<String>>`），7.x 去掉了该继承关系，不再有 Map 语义。

**兼容处理**：

新增 `WebHttpHeaders.java` — 继承 `HttpHeaders` 并实现 `MultiValueMap<String, String>`，内部通过 `MethodHandle` 桥接 Spring 7.x 的 `asMultiValueMap()` 获取父类委托，Spring 6.x 下直接走父类继承。创建 `HttpHeaders` 的地方改为 `new WebHttpHeaders()`：

| 文件 | 修改 |
|------|------|
| `WebHttpHeaders.java` | **新建** — 跨版本兼容的 MultiValueMap 实现 |
| `BaseWebServerHttpResponse.java` | `new HttpHeaders()` → `new WebHttpHeaders()` |
| `NettyServerHttpRequest.java` | `new HttpHeaders()` → `new WebHttpHeaders()` |

**原理**：`HttpHeaders.get(key)` 在 6.x 和 7.x 都存在。6.x 中 `HttpHeaders implements MultiValueMap<String, String>`，可直接作为 Map 使用；7.x 需要通过 `asMultiValueMap()` 获取内部委托。`WebHttpHeaders` 在构造时通过 `HEADERS_IS_MULTI_VALUE_MAP` 静态常量判断当前版本（JIT 消除死分支），7.x 下借助 `MethodHandle` 反射调用 `asMultiValueMap()` 获取委托引用。

**适配模式**：子类化 + 反射桥接，运行时 JIT 消除死分支，零性能损耗。

---

### 11.2 `ResponseStatusException.getResponseHeaders()` → `getHeaders()`

**Spring 7.x 变更**：`ResponseStatusException.getResponseHeaders()` 重命名为 `getHeaders()`。

**兼容处理**：
- 新建 `ResponseStatusExceptionAdapter.java` — 反射适配器，运行时检测 `getHeaders()` 方法是否存在
- `ResponseStatusExceptionResolver.java` — 通过适配器调用，而非直接调用

**适配模式**：反射桥接，运行时检测目标方法名。

---

### 11.3 `HttpMessageNotReadableException` 构造函数签名变更

**Spring 7.x 变更**：构造函数新增 `HttpInputMessage` 参数。

**兼容处理**：

| 文件 | 修改 |
|------|------|
| `HttpBodyCodecRegistry.java` | `new HttpMessageNotReadableException(msg)` → `new HttpMessageNotReadableException(msg, null, request)` |
| `RequestBodyResolver.java` | `new HttpMessageNotReadableException(msg, cause)` → `new HttpMessageNotReadableException(msg, cause, request)` |

**原理**：三参构造函数 `HttpMessageNotReadableException(String, Throwable, HttpInputMessage)` 在 Spring 6.x 就已存在，7.x 移除了旧的两参版本。传递 `request` 作为第三个参数在两端都合法。

---

### 11.4 MediaType 常量删除

**Spring 7.x 变更**：删除 `MediaType.APPLICATION_STREAM_JSON` 和 `MediaType.APPLICATION_JSON_UTF8`。

**兼容处理**：

新增 `MediaTypeUtils.java` — 公共工具类，集中存放跨版本 MediaType 常量和工具方法。

| 文件 | 修改 |
|------|------|
| `MediaTypeUtils.java` | **新建** — 定义 `APPLICATION_STREAM_JSON = new MediaType("application", "stream+json")` |
| `ReactiveReturnValueResolver.java` | `new MediaType("application", "stream+json")` → `MediaTypeUtils.APPLICATION_STREAM_JSON` |
| `StreamJsonEmitter.java` | `new MediaType("application", "stream+json")` → `MediaTypeUtils.APPLICATION_STREAM_JSON` |
| `HttpBodyCodecRegistry.java` | `APPLICATION_JSON_UTF8` → `APPLICATION_JSON` |
| 测试文件 | 同上替换 |

**原理**：`APPLICATION_JSON_UTF8` 自 Spring 5 起已弃用（JSON 默认 UTF-8）；`APPLICATION_STREAM_JSON` 集中定义在 `MediaTypeUtils` 中，避免散落 `new MediaType("application", "stream+json")`。

---

### 11.5 MediaType 排序方法删除

**Spring 7.x 变更**：删除 `MediaType.sortBySpecificity()`、`sortBySpecificityAndQuality()`、`SPECIFICITY_COMPARATOR`。

**兼容处理**：

| 文件 | 修改 |
|------|------|
| `MediaTypeUtils.java` | **新建** — 定义 `SPECIFICITY_COMPARATOR`、`QUALITY_AND_SPECIFICITY_COMPARATOR`、`sortBySpecificity()`、`compareSpecificity()` |
| `HttpBodyCodecRegistry.java` | 内联 `compareSpecificity()` → 委托 `MediaTypeUtils.compareSpecificity()` |

**适配模式**：Comparator 逻辑集中到 `MediaTypeUtils` 工具类，各调用点统一引用。注意 wildcard 优先级需要比 `"json" > "*"` 的字典序更优先，末尾补充 `type.compareToIgnoreCase` + `subtype.compareToIgnoreCase` 兜底。

---

### 11.6 `UriComponentsBuilder.fromOriginHeader()` 删除

**Spring 7.x 变更**：移除该方法。

**兼容处理**：`CorsUtils.java` — `fromOriginHeader(origin)` → `fromUri(URI.create(origin))`

**原理**：`fromUri(URI.create())` 在 6.x 和 7.x 都存在，语义相同。

---

### 11.7 `ListenableFuture` 完全删除

**Spring 7.x 变更**：移除 `org.springframework.util.concurrent.ListenableFuture` 及相关回调类。

**兼容处理**：
- 新建 `ListenableFutureAdapter.java` — 完全反射实现，运行时检测类是否存在
- `ListenableFutureReturnValueResolver.java` — 增加 null guard，`supportsReturnType` 在类不存在时返回 `false`
- 测试改为反射构造实例 + `assumeTrue(isAvailable())` 守卫

**适配模式**：全反射桥接，运行时判断可用性。测试中用 `assumeTrue` 跳过不可用版本。

---

### 11.8 `MethodParameter` 自动设置 ParameterNameDiscoverer

**Spring 7.x 变更**：`MethodParameter` 构造器自动绑定 `DefaultParameterNameDiscoverer`，配合 `-parameters` 编译标志可直接获取参数名。

**兼容处理**：`MethodArgContextTest.java` — fallback 名称测试中调用 `mp.initParameterNameDiscovery(null)` 清理 discoverer。

**原理**：6.x 下 `getParameterName()` 在无 discoverer 时返回 `null`，触发 fallback 逻辑；7.x 自动发现参数名 `"name"`，需要清理才能测试 fallback 路径。

---

### 11.9 `HttpEntity` 构造函数签变更

**Spring 7.x 变更**：`HttpEntity(T body, MultiValueMap<String, String> headers)` → `HttpEntity(T body, HttpHeaders headers)`。`null` 传参时构造函数二义。

**兼容处理**：`HttpEntityReturnValueResolverTest.java` — `new HttpEntity<>("body", null)` → `new HttpEntity<>("body", new HttpHeaders())`

---

### 11.10 POM Profile 配置

在 `pom.xml` 中添加两个新 profile：

```xml
<profile>
    <id>spring-boot-4.0</id>
    <properties>
        <spring-boot.version>4.0.7</spring-boot.version>
    </properties>
</profile>
<profile>
    <id>spring-boot-4.1</id>
    <properties>
        <spring-boot.version>4.1.0</spring-boot.version>
    </properties>
</profile>
```

**使用方式**：
```bash
# 默认 SB 3.2.x
mvn test
# SB 4.0.x
mvn test -Pspring-boot-4.0
# SB 4.1.x
mvn test -Pspring-boot-4.1
```

### 11.11 已知注意事项

1. **切换 Profile 必须 `clean`**：不同 Spring 版本对 `HttpHeaders.get(name)` 生成不同字节码（6.x → `invokeinterface Map.get(Object)`，7.x → `invokevirtual HttpHeaders.get(String)`），不 clean 会加载旧 class 文件导致 `NoSuchMethodError`。
2. **测试必须在所有 Profile 上通过**：编译通过不意味着运行正确。例如 `containsKey` 在 6.x 返回 `boolean`、在 7.x 返回 `List<String>`，Mockito mock 行为完全不同。
3. **`spring-boot-4.1` profile 使用 SB 4.1.0**，对应 Spring Framework 7.1.x，保持与 4.0.x 相同的兼容处理。
