# Performance Principles

This project's performance does not come from a vague claim like "Netty is faster than Tomcat." Netty is just an event-driven I/O framework — true performance comes from engineering tradeoffs at the framework design level.

The following is ordered **by impact, largest to smallest** — techniques at the top benefit every request, while those at the bottom apply only to specific scenarios.

## 1. Startup Pre-caching, Zero Reflection & Matching at Runtime ← Core

### How

The framework uses a three-phase lifecycle (`initComponentPhase1/2/3`) to resolve and cache all metadata at startup. Each Registry iterates all Mappings in Phase3, computes the corresponding strategy, and writes to `MappingCacheKey`.

`MappingCacheKey` uses **integer array indices** (instead of `ConcurrentHashMap`):

```java
// MappingCacheKey internals
final int index;  // Fixed index assigned at startup

// Access is direct array lookup
mappingContext.get(MAPPING_CACHE_KEY)
    → handlerMethod.methodCache[cacheKey.index]  // O(1), no locks, no hashing
```

All of the following components use this mechanism for startup pre-matching:

| Component | Runtime Behavior | Spring MVC Comparison |
|-----------|-----------------|----------------------|
| `ArgumentResolverRegistry` | Directly calls cached resolver | Per-request resolver iteration + `synchronized` cache |
| `ReturnValueResolverRegistry` | Directly calls cached resolver | Per-request iteration of return value handlers |
| `BizPoolRegistry` | Array index to thread pool ref | — |
| `InterceptorRegistry` | Returns pre-matched interceptor array | Per-request path matching for interceptors |
| `WebDataBinderRegistry` | Returns pre-cached Validator | Per-request `@InitBinder` scanning |
| `HttpBodyCodecInterceptorRegistry` | Returns pre-cached Advice chain | Per-request `RequestBodyAdvice` matching |

### Why This is the Core Technique

- **Request-type independent**: whether CPU or IO intensive, pre-caching benefits every request
- **Eliminates synchronization**: Spring MVC's `HandlerMethodArgumentResolverComposite` uses `synchronized` blocks for resolver caching — lock contention under high concurrency
- **Array access is 3-5x faster than Map**: `MappingCacheKey.index` directly indexes `Object[]`, eliminating `ConcurrentHashMap.get()`'s 4 volatile reads and hashCode computation
- **Compile-time vs runtime matching**: startup matching = one-time cost; runtime matching = per-request recurring cost

## 2. Bytecode Generation Replaces Reflection

### How

Two levels of invocation optimization:

**Level 1 (`@Optimize` annotation) → ASM bytecode generation**

`FastInvokerGenerator` uses Spring ASM (`ClassWriter`) at startup to generate dedicated Java classes for each `@Optimize`-annotated controller method. The generated bytecode uses the `INVOKEVIRTUAL` instruction directly:

```java
// Generated code is equivalent to:
public class HelloController_sayHello_0 implements Invoker {
    private final HelloController target;

    public Object invoke(Object[] args) {
        return target.sayHello((String) args[0]);  // INVOKEVIRTUAL
    }
}
```

`INVOKEVIRTUAL` is the JVM's most fundamental invocation instruction — JIT can inline it directly, equivalent to handwritten code.

**Level 2 (default) → MethodHandle**

```java
// MethodHandleInvoker.java line 27
return methodHandle.invokeExact(args);
```

`MethodHandle.invokeExact()` is a JVM intrinsic — JIT can inline it as a direct call, eliminating `Method.invoke()`'s access checks and argument boxing.

### Why It's Faster

| Invocation Type | Approximate Cost | Overhead Source |
|----------------|-----------------|-----------------|
| `INVOKEVIRTUAL` direct call | ~10ns | None |
| `MethodHandle.invokeExact()` | ~30ns | No access check |
| `Method.invoke()` | ~200ns | access check + varargs boxing + type validation |

> Note: this benefit is independent of which thread executes the call — even when dispatched to a business thread pool, the invocation is still `INVOKEVIRTUAL`/`MethodHandle`.

## 3. O(1) Routing — Multi-Level Optimizer Chain

### How

`MappingRegistry.optimizeMapping()` buckets and multi-level optimizes the route table at startup:

**Step 1: Bucketing**

```
No-wildcard paths (/api/user/list)    → simpleUrlList    (HashMap.get  O(1))
Single-level wildcard (/api/user/{id}) → simpleWildcardList (prefix/suffix index)
Full wildcard (/**)                   → fullWildcardList  (iteration)
```

**Step 2: Build optimizer chain**

```
FullPathRouterOptimizer  (exact path → HashMap.get(path)  O(1))
  ↓ miss
PrefixPathRouterOptimizer (prefix match → HashMap.get(prefix) O(1))
  ↓ miss
SuffixPathRouterOptimizer (suffix match → HashMap.get(suffix) O(1))
  ↓ miss
LoopPathPatternRouterOptimizer (iterate PathPatternRouter[]  O(n))
```

### Why It's Faster

- Spring MVC runs `AntPathMatcher.match()` for every path regardless of wildcards
- 90%+ of this framework's paths are exact — a single `HashMap.get()` suffices
- The statistical prefix optimizer automatically selects the optimal prefix depth, minimizing wildcard path traversal scope

This benefit applies to every request.

## 4. Avoid Implicit Object Creation — GC-Friendly

### How

The design principle is explicitly stated in the code standards: **no `new ArrayList`, no boxing, no temporary object creation on the request path.**

Specific implementations:
- `BaseWebServerHttpRequest.fastAttributes[]`: `Object[]` replaces `ConcurrentHashMap`, zero String object creation
- `NettyServerHttpResponse.getBuf()`: `ByteBuf` deferred until first write operation
- `BaseWebServerHttpResponse.setCommitted()`: `AtomicBoolean` CAS, executes only once per request lifecycle
- All Registries build their lists at startup; runtime read operations create no temporary collections

### Why It's Faster

GC pressure directly translates to STW pauses. Reducing 1-2 short-lived object allocations per request translates to 1.7 million fewer allocations per minute at 28K ops/s.

## 5. ByteBuf Reference Counting — Precise Memory Management

### How

`NettyServerHttpRequest` implements `acquire()` / `release()` for ByteBuf lifecycle management across thread switches (EventLoop → business thread pool):

```java
// DispatcherHandler lines 86-98
req.acquire();         // retain FullHttpRequest
try {
    executor.execute(() -> {
        try { processRequest(...); }
        finally { req.release(); }  // Business thread releases after processing
    });
} catch (RejectedExecutionException e) {
    req.release();      // Release even when thread pool is full
    throw e;
}
```

Large bodies (>4KB) use `retainedDuplicate()` zero-copy, avoiding extra heap memory allocation.

### Why It's Faster

- Direct `DirectByteBuf` operations: eliminates heap→direct copy
- Reference counting precisely controls direct memory release, independent of GC (`Cleaner` recovery has latency)
- `Unpooled.wrappedBuffer(data)` zero-copy wraps byte[]

## 6. EventLoop Direct Processing with Reactive Programming & Virtual Threads

### How

In `DispatcherHandler.handleWithFullMatch()`, if the controller does **not** have `@RunInPool`, the method executes on the `default` business thread pool by default (configured via `pool.*` properties). Lightweight endpoints can explicitly switch to EventLoop via `@RunInPool(RunInPool.EVENTLOOP)`, or globally via `pool.default-execute-mode=eventloop`.

### Key Understanding: It's Not "Optimizing for You" — It's "Giving You Choice"

Traditional Servlet containers (Tomcat) force all requests through the container thread pool — business code cannot choose where it executes. This framework allows requests to execute directly on EventLoop, with the business deciding whether to switch to a thread pool — the real value is **freedom of programming model choice**:

**Option 1: Synchronous blocking (default)**

Without `@RunInPool`, methods execute on the `default` business thread pool (configured via `pool.*`). Use `@RunInPool("custom")` to dispatch to a custom thread pool.

**Option 2: Reactive programming (`@RunInPool(RunInPool.EVENTLOOP)`)**

```java
@GetMapping("/reactive")
@ReactiveSupport(highWaterMark = 256, lowWaterMark = 64)
public Publisher<Data> reactiveEndpoint() {
    return Flux.range(1, 1000)
            .map(i -> new Data(i, "item-" + i));
}
```

Business uses reactive drivers (R2DBC, WebClient, Reactive Redis) — all IO is non-blocking, **executing directly on EventLoop without occupying business threads**. The framework adapts `Publisher` to streaming output or `DeferredResult` via `ReactiveReturnValueResolver`, with backpressure parameters (`highWaterMark`/`lowWaterMark`) controlling memory levels.

This approach is particularly effective for high-concurrency, IO-intensive workloads with scattered latency (e.g., gateways, aggregation services) — EventLoop's efficient scheduling + reactive library async drivers, no business thread pool needed.

**Option 3: JDK 21 virtual threads (future)**

```java
@GetMapping("/future-ready")
@RunInPool(RunInPool.EVENTLOOP) // Virtual threads + EventLoop non-blocking switching
public Result<Data> query() {
    return Result.ok(repository.findById(1L));
}
```

This framework's synchronous programming model is naturally compatible with virtual threads. When JDK 21+ virtual threads become mainstream, `@RunInPool` + `EventLoop` allows virtual threads to execute blocking operations directly on EventLoop — virtual thread `park`/`unpark` means blocking no longer blocks OS threads, eliminating the need for a separate business thread pool.

### Current Benefit vs Architectural Significance

Today, direct EventLoop processing saves thread switching (~1-3μs) for pure CPU endpoints — a nice side benefit. **Architecturally more important**: the framework doesn't lock into a single model — the business can choose synchronous blocking, reactive, or future virtual threads without switching web frameworks.

## 7. Lock-free Drain Loop — SSE/Streaming Throughput

### How

`NettyStreamSender` uses `MpscUnboundedArrayQueue` + `AtomicInteger wip` to form a lock-free Drain Loop:

```java
private final MpscUnboundedArrayQueue<ByteBuf> queue;  // Multi-producer single-consumer lock-free queue

void drain() {
    int missed = 1;
    for (;;) {
        while (channel.isWritable()) {
            ByteBuf buf = queue.poll();
            channel.write(new DefaultHttpContent(buf));
        }
        channel.flush();
        missed = wip.addAndGet(-missed);
        if (missed == 0) break;
    }
}
```

### Scope

SSE and streaming response only. In benchmarks, perf SSE throughput reaches 3.89x of Spring MVC (4 threads), scaling to 6.64x under high concurrency.

## 8. Netty-Level Transport Optimizations

- `TCP_NODELAY=true`: disables Nagle's algorithm, friendly to small-packet latency
- `WriteBufferWaterMark`: precise backpressure thresholds
- `DefaultFileRegion`: sendfile zero-copy file transfer
- HTTP/2 multiplexing (optional)

These are general benefits shared by all Netty-based frameworks, not unique to this project.

## Summary

| Priority | Technique | Impact | Quantified Estimate |
|----------|-----------|--------|---------------------|
| ★★★★★ | Startup pre-caching, zero matching at runtime | 100% | Eliminates per-request argument/return value resolver iteration |
| ★★★★★ | ASM/MethodHandle replaces reflection | 100% | Method invocation from ~200ns down to ~10-30ns |
| ★★★★★ | O(1) HashMap routing | 100% | Path matching from O(n) down to O(1) |
| ★★★★☆ | GC-friendly, no temporary allocation | 100% | Millions fewer allocations per minute |
| ★★★★☆ | ByteBuf zero-copy | 100% | Eliminates 1-2 data copies |
| ★★★★☆ | EventLoop direct processing | Optional — three models | Foundation for reactive / virtual threads |
| ★★☆☆☆ | Lock-free Drain Loop | SSE only | 3.89x(4t) / 6.64x(64t) in SSE scenarios |
| ★★☆☆☆ | Netty transport optimization | Marginal | Shared by all Netty frameworks |

**Core conclusion**: the framework eliminates all runtime "find" and "match" overhead through **startup-time deterministic resolution** — this is the primary performance driver. Bytecode generation to eliminate reflection further reduces method invocation cost (~200ns → ~30ns). The default `default` business thread pool means most IO handlers don't need annotations; lightweight endpoints can explicitly switch to EventLoop via `@RunInPool(RunInPool.EVENTLOOP)`. This implements the "business chooses the programming model" design philosophy — the framework provides flexible control without mandating a single model.
